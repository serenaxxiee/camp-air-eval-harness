Shared Vocabulary

The building blocks of every evaluation

TermDefinitionExample
Test CaseA single prompt + expected behavior pair. The atomic unit of evaluation.“How many PTO days?” → correct policy + citation
Acceptance CriteriaWhat “good” looks like. Specific enough that two people agree on pass/fail.“Must state 15 days and cite the Handbook”
Quality SignalA named dimension of quality derived from patterns. Agent-specific, not generic.Policy Accuracy, Source Attribution, Escalation Judgment
GraderApplies criteria to a response. Deterministic (regex, length) or model-graded (LLM judge).keywordMatch("hand wash"), modelGraded("empathetic")
Eval SetA collection of test cases measuring quality dimensions. Your reusable, growing asset.20 cases across core, variations, architecture, edge
HarnessThe engine that runs test cases, applies graders, and produces a results report.Copilot Studio → 7/12 passed (58%)

7 Test Methods in Copilot Studio

Choose based on what you're checking — use two methods per test case where possible

MethodTypeWhat It ChecksBest For
Keyword MatchDeterministicSpecific words/phrases appear (All or Any mode)Policy compliance, must-include checks
Compare MeaningModel-basedSemantic similarity to expected answerFactual accuracy, flexible phrasing
Text SimilarityDeterministicCosine similarity score (0–1)Verbatim compliance, legal text
Exact MatchDeterministicCharacter-for-character equalityClassification labels, routing codes
General QualityModel-basedLLM judges: Relevance, Groundedness, Completeness, AbstentionTone, helpfulness, open-ended quality
Tool UseDeterministicAgent triggered correct tools/topicsTool invocation, source selection, routing
CustomModel-basedYour criteria + labels (pass/fail)Domain-specific checks (compliance, triage)

/eval-faq

Your always-on eval methodology expert

What it does

Answers any eval methodology question with opinionated, source-cited guidance. Grounded primarily in Microsoft’s eval ecosystem — MS Learn, Eval Scenario Library, Triage Playbook — supplemented by select industry sources.

Primary sources

• MS Learn — 4-stage framework, 7 test methods
• Eval Scenario Library — 80 scenarios, quality signals
• Triage Playbook — SHIP/ITERATE/BLOCK, root causes
• Camp AIR harness — CLI flags, fields, thresholds

Example prompts
/eval-faq What's the difference between
deterministic and model-graded graders?

/eval-faq How many eval cases do I need
for a workshop?

/eval-faq My grader gives false failures
on case-003, how do I debug it?
Key behavior

3–5 sentence answers. Specific numbers. No “it depends” without a concrete recommendation. Cites sources at the end.

Available throughout the entire session — use it whenever you have a question about eval methodology

/eval-suite-planner

Plan your eval suite before writing a single test case — grounded in Microsoft’s Eval Scenario Library + MS Learn

What it does

Takes a plain-English agent description and uses the Eval Scenario Library to match your agent to the right business-problem and capability scenario types. Outputs a scenario plan table that feeds directly into /eval-generator.

Output → handoff to /eval-generator
SectionContent
Scenario plan tableNamed scenarios, tags, evaluation methods
Quality signalsPolicy Accuracy, Source Attribution, Personalization, Action Enablement, Privacy
ThresholdsRisk-based: ≥90% core, ≥95% safety, ≥70% edge
Workshop shortcut6–8 highest-signal cases for a 15-minute exercise
Example
/eval-suite-planner I am building an
email triage agent that reads incoming
emails and labels them urgent,
not-urgent, or spam. It should never
label a real customer email as spam.
Key behavior

Specific, not generic. Names actual scenarios and maps each to evaluation methods (Keyword Match, Compare Meaning, Tool Use, General Quality). The scenario plan table is the blueprint for /eval-generator.

/eval-generator

Generate test cases from your plan — or from scratch

Two modes

Plan mode: Reads the scenario plan table from /eval-suite-planner in the conversation. Generates one test case per scenario — preserving scenario names, tags, and evaluation methods.

Standalone: Accepts a plain-English agent description and generates 6–8 cases from scratch.

Two output formats

• Copilot Studio table (primary) — Question, Expected Response, Test Method, Pass Score

• CSV + docx (secondary) — for offline review and sharing

Smart guard

Vague description (<2 sentences)? Asks exactly one clarifying question. No plan in conversation? Asks you to run /eval-suite-planner first.

Example — plan mode
/eval-suite-planner I'm building a
customer support agent...
[planner outputs scenario plan table]

/eval-generator
→ generates from plan, one case per row
Output includes

• Copilot Studio test set table (copy-paste ready)

• CSV + docx output for offline review and sharing

• Reviewer notes: hardest case, one more to add, human review reminder

/eval-result-interpreter

Triage eval results using Microsoft’s Triage & Improvement Playbook

What it does

Reads a Copilot Studio CSV export (or pasted results) and produces a structured triage report grounded in the Triage Playbook’s 4-layer framework: score interpretation, failure triage, remediation mapping, and pattern analysis.

SHIP / ITERATE / BLOCK decision tree
VerdictCondition (Triage Playbook)
SHIPAll signals above risk-based thresholds
SHIP WITH GAPSCore passing, capability gaps documented
ITERATECore business below threshold, targeted fixes needed
BLOCKSafety/compliance failure OR <60% overall

100% on first run = red flag. Add harder cases.

Example — CSV input
/eval-result-interpreter
C:\Downloads\Evaluate Agent.csv

→ Reads CSV: 5/9 passed (55.6%)
→ Verdict: BLOCK
→ Pattern: 3 failures share
  "Question not answered"
3 root cause types (Triage Playbook)

Every failure classified as one of:

• Eval Setup Issue — test/grader is wrong, not the agent (20%+ of new eval failures)

• Agent Config Issue — agent produced a bad response, fix prompt/tools/knowledge

• Platform Limitation — can’t fix via config, document and work around

Hands-On Part 1: Explore + Plan 10 min

Step 1 — Warm up with /eval-faq
/eval-faq How many test cases do I need for a workshop eval?

Step 2 — Plan your eval suite

Pick an agent (your own Copilot Studio agent, or one of these examples):

/eval-suite-planner I am building a customer support agent that answers
questions about our product return policy and helps customers submit
return requests. It should follow the return policy and never promise
things the policy does not allow.
Done when

You have a scenario plan table in your Claude conversation with named scenarios, tags, and evaluation methods. Do NOT close this session.

Hands-On Part 2: Generate 15 min

Run this in the same session as Part 1 — the generator reads your plan automatically

Step 1 — Generate test cases from your plan
/eval-generator

The generator detects the planner output and produces one test case per scenario. Output: Copilot Studio table ready for upload.

Step 2 — Review and refine

• Read each test case. Delete any that would never happen in production.

• Add at least one test case of your own (an edge case or adversarial scenario you know about).

• Check that adversarial cases test something meaningful (prompt injection, out-of-scope request).

Step 3 — Save your test set

Copy the Copilot Studio table from the generator output. This is your complete eval test set, ready for upload to Copilot Studio when you're back at your desk.

Done when

You have 6–8 reviewed test cases in a Copilot Studio table format. Save the scenario plan table as a reference — CSV exports lose this metadata.

Hands-On Part 3: Interpret + Triage 15 min

Step 1 — Get the synthetic results CSV

We provide a pre-built Copilot Studio eval results CSV with realistic pass/fail data. This is the same format you'll get when you run your own test set in Copilot Studio later.

synthetic-eval-results.csv   # provided in the repo
Step 2 — Interpret results
/eval-result-interpreter synthetic-eval-results.csv
Step 3 — Reflect (3 debrief questions)

1. What passed? Does it make sense, or are your tests too easy?

2. For each failure: is it an eval setup issue (grader/test is wrong) or an agent issue (agent needs fixing)?

3. What is the #1 thing you would change before re-running?

Done when

You have a SHIP / ITERATE / BLOCK verdict and know whether your failures are eval problems or agent problems. If 100% passed, your eval is too easy — add harder cases.

Key Takeaways

Start with criteria, not code.
Write what “good” looks like in plain English before touching any eval code.
Small datasets are fine. Representative ones are essential.
10 well-chosen cases beat 100 random ones. Include happy paths, edge cases, and adversarial.
AI accelerates the scaffolding, not the thinking.
Use /eval-generator for speed. But you define criteria, review cases, and interpret results.
Eval sets compound over time.
Every incident, every edge case, every lesson learned becomes a test case that protects you forever.
