# Camp AIR ‚Äî Eval Harness

This harness is **step 3** of the 4-skill Camp AIR eval lifecycle: `/eval-suite-planner` plans your suite (categories, counts, graders, thresholds); `/eval-generator` produces dataset JSON from that plan; **this harness** runs each case against your agent and scores the outputs; `/eval-result-interpreter` reads the results and returns a SHIP / ITERATE / BLOCK verdict with triage notes.

## Skill lifecycle

The 4-skill Camp AIR eval pipeline:

```
1. /eval-suite-planner   ‚Üí Plan your eval suite (categories, counts, thresholds)
2. /eval-generator       ‚Üí Generate scenario JSON (save to my-evals/datasets/)
3. npm run eval:workshop ‚Üí Run the harness against your scenarios
4. /eval-result-interpreter ‚Üí Paste the summary above to get SHIP/ITERATE/BLOCK verdict
```

## Quick Start

```bash
git clone <repo-url>
cd eval-harness
npm install

# Verify everything works (requires ANTHROPIC_API_KEY ‚Äî see below)
npm test
```

`npm test` runs the demo summarizer eval and should print a result table with
pass/fail rows. A score of 4/6, 5/6, or 6/6 is normal ‚Äî the LLM-graded grader
has natural variance.

## API key setup

```bash
# macOS / Linux
export ANTHROPIC_API_KEY=sk-ant-...

# Windows PowerShell
$env:ANTHROPIC_API_KEY="sk-ant-..."
```

Both the model under test (`run()`) and the model-graded grader require the key.
The deterministic graders (`lengthAndTopicGrader`, `keywordPresenceGrader`,
`jsonOutputGrader`) never call the API.

## Your first eval

```bash
# 1. See output immediately ‚Äî runs the demo eval
npm test

# 2. Run your participant workspace eval
npm run eval -- --file my-agent-eval.ts
```

The `--file` flag accepts a bare filename or a relative path. Bare filenames are
searched in `my-evals/` then `examples/`, matched by filename. A relative path
(e.g. `examples/summarizer-eval.ts`) is resolved from the harness root directly.

## CLI flags

| Flag | Description |
|------|-------------|
| `--file <path>` | Eval file to run (required). Bare filename or relative path. |
| `--ci` | Exit code 1 if any cases fail (for CI pipelines) |
| `--tags <tag>` | Only run cases whose `tags` array includes this value |
| `--verbose` | Print the full model output under each case row |

```bash
npm run eval -- --file my-eval.ts
npm run eval -- --file my-eval.ts --verbose
npm run eval -- --file my-eval.ts --tags edge-case
npm run eval -- --file my-eval.ts --tags adversarial --ci
```

## Writing an eval

`my-evals/my-agent-eval.ts` is the pre-populated participant workspace ‚Äî it runs
out of the box and is ready for you to customise.

1. Open `my-evals/my-agent-eval.ts` ‚Äî it already exists with placeholder cases
   and TODO comments. Follow the **TODO comments** top-to-bottom.
2. Run at any point: `npm run eval -- --file my-agent-eval.ts`
3. Add more cases covering all four scenario types: `happy-path`, `edge-case`,
   `adversarial`, `capability`.

See `examples/` for complete working examples:
- `summarizer-eval.ts` ‚Äî single-turn summarization (main workshop demo)
- `multi-turn-eval.ts` ‚Äî multi-turn conversation with a custom grader
- `starter-eval.ts` ‚Äî clean reference template; copy it to start a second eval from scratch

## Built-in graders

Use `autoGrader()` if your dataset was generated by `/eval-generator` ‚Äî it handles all `expected` fields automatically. Use individual graders when you need fine-grained control.

| Grader | What it checks | `expected` fields read |
|--------|---------------|------------------------|
| `autoGrader()` | Inspects each case's `expected` block and automatically applies the right sub-graders. **Recommended for datasets from `/eval-generator`.** | All `expected` fields (dispatches automatically) |
| `lengthAndTopicGrader()` | Output <= max chars; contains required topic | `max_length`, `must_include_topic` |
| `keywordPresenceGrader()` | All required keywords present (case-insensitive) | `must_include_keywords` |
| `contentExclusionGrader()` | Output does NOT contain a forbidden string (case-insensitive); pass when field absent | `must_not_contain` |
| `formatGrader()` | Output matches required format: `"json"`, `"bullet-list"`, or `"plain-text"`; pass when field absent | `format` |
| `classificationGrader()` | Output contains the expected classification label (case-insensitive); pass when field absent | `classification` |
| `minItemsGrader()` | Output has at least N non-empty lines; pass when field absent | `min_items` |
| `jsonOutputGrader(keys?)` | Output is valid JSON; optional required top-level keys | (none ‚Äî keys in code) |
| `modelGradedGrader(criteria)` | LLM judges output against criteria string | (none ‚Äî criteria in code) |

## Project layout

```
src/           Harness internals (types, runner, graders, reporter, CLI)
examples/      Working example evals and their datasets
my-evals/      Your workspace ‚Äî edit my-agent-eval.ts to get started
datasets/      Your eval datasets (gitignored ‚Äî safe to put real inputs here)
results/       Auto-generated JSON result files (gitignored)
```

Note: `results/` and `datasets/` are in `.gitignore` ‚Äî JSON files written there will not be committed. The `datasets/README.md` file is kept tracked as a reference; all other content in `datasets/` is ignored.
