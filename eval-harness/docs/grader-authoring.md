# Grader Authoring Guide

A grader is any function that matches the `Grader` type from `src/types.ts`:

```typescript
type Grader = (
  output: string,
  expected: Record<string, unknown>,
  input: Record<string, unknown>
) => Promise<GraderResult>;
```

`output` is the raw string your agent returned. `expected` and `input` come
directly from the dataset case.

---

## Quick start: autoGrader (recommended for workshop)

If your dataset was generated by `/eval-generator`, use `autoGrader()` and
you're done — no grader configuration needed. It inspects each case's `expected`
block and automatically runs the right sub-graders.

```typescript
import { autoGrader } from "../src/graders.js";

graders: [autoGrader()]
```

The sub-graders it dispatches to:

- `expected.max_length` or `expected.must_include_topic` — length and topic check
- `expected.must_include_keywords` — keyword presence check
- `expected.must_not_contain` — content exclusion check
- `expected.format` — format check (`"json"`, `"bullet-list"`, `"plain-text"`)
- `expected.classification` — classification label check
- `expected.min_items` — minimum line count check
- `expected.criteria` — model-graded score (requires `ANTHROPIC_API_KEY`)

If none of these fields are present on a case, `autoGrader` passes
unconditionally — so always include at least one `expected` field per case.

Read on if you want to write custom graders or mix individual graders.

---

## When to use each type

| Grader type | Use when |
|-------------|----------|
| Deterministic | You can express "correct" as a rule (length, keywords, regex, JSON schema) |
| Model-graded | You need qualitative judgment — tone, relevance, safety, instruction-following |

Default to deterministic graders where possible. They are fast, free, and
produce consistent results. Add a model-graded grader when deterministic checks
are not sufficient to catch regressions you actually care about.

---

## Writing a deterministic grader

Return a factory function that returns the `Grader`. Read fields you need from
`expected`; ignore fields you do not recognise.

```typescript
import type { Grader, GraderResult } from "../src/types.js";

// Passes when the output contains no ALL-CAPS words (shouting check).
export function noShoutingGrader(): Grader {
  return async (output: string): Promise<GraderResult> => {
    const shouting = /\b[A-Z]{3,}\b/.test(output);
    return {
      name: "no-shouting",
      passed: !shouting,
      reason: shouting ? "Output contains ALL-CAPS word(s)." : "No shouting detected.",
    };
  };
}
```

Add it to your eval's `graders` array: `noShoutingGrader()`.

## Writing a model-graded grader

The built-in `modelGradedGrader` covers most cases. Pass a precise
`criteriaDescription` string — this is injected directly into the grader
prompt, so write it as one or two complete sentences that describe what a
passing response looks like.

```typescript
import { modelGradedGrader } from "../src/graders.js";

modelGradedGrader(
  "The response answers the user's question with at least one specific fact " +
  "from the provided context. It must not invent information not present in " +
  "the context, and must not refuse to answer if the context is sufficient."
)
```

The grader model returns a score from 0.0 to 1.0. The grader passes when the
score is >= 0.6. That threshold is defined in `src/graders.ts` as
`MODEL_GRADED_PASS_THRESHOLD`.

If you need a different threshold or a different grader prompt structure, write
a custom grader that calls the Anthropic client directly, following the same
pattern as `modelGradedGrader` in `src/graders.ts`.

## GraderResult fields

| Field | Required | Notes |
|-------|----------|-------|
| `name` | Yes | Short label shown in the result table |
| `passed` | Yes | `true` = this grader considers the output acceptable |
| `score` | No | 0.0–1.0 continuous score; omit for deterministic graders |
| `reason` | No | One-line explanation shown in the result table |
