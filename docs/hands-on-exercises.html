Camp AIR — AI Agent Evaluation: Hands-On Exercises

Session block: 0:25–0:50 (25 minutes total) Format: Work individually or in pairs. A facilitator is circulating — raise your hand if you are stuck.

Quick-Reference Cheat Sheet

Keep this visible while you work.

Key commands

# Run your eval
npm run eval -- --file my-eval.ts

# Run in CI mode (exit code on failure)
npm run eval -- --file my-eval.ts --ci

# Filter by tag
npm run eval -- --file my-eval.ts --tags edge-case

# Verbose output (full model responses)
npm run eval -- --file my-eval.ts --verbose

Dataset case format

{
  "id": "case-001",
  "input": {
    "text": "..."
  },
  "expected": {
    "max_length": 400,
    "must_include_topic": "revenue"
  },
  "tags": ["happy-path", "financial"]
}

Three debrief questions (answer before 0:50)

  1. Which cases passed? Which failed?
  2. Does the failure make sense, or does it look like the grader is wrong?
  3. What is one thing you would change — in the prompt, the dataset, or the grader?

Stage A — Explore + Plan with /eval-faq and /eval-suite-planner

Time box: 10 minutes (0:20–0:30)

You have just watched the live demo. Before you build anything, use /eval-faq to internalize the methodology, then use /eval-suite-planner to create a plan for your agent. Stay in this same Claude Code session for Stages A and B — the generator in Stage B reads the planner's output from your conversation.

Steps

1. Open a new Claude Code session in your terminal.

claude

2. Verify skills are installed. Type /eval-faq what is eval — if you see a structured answer, skills are working. If not, check that you copied the skill folders to ~/.claude/skills/ and restarted Claude Code.

3. Run /eval-faq and ask at least two of the questions below. Copy them verbatim or rephrase them in your own words.

/eval-faq What eval scenarios should I use for a RAG agent?

/eval-faq How do I interpret a 75% knowledge grounding score?

/eval-faq What is the difference between business-problem and capability scenarios?

/eval-faq When should I use a model-graded grader instead of a deterministic one?

/eval-faq What makes a good adversarial test case?

/eval-faq How many cases do I need in a dataset to get meaningful signal?

4. As you read each answer, look for: - The names of specific scenario types or grader types mentioned - Any thresholds or rules of thumb (numbers, percentages, counts) - Anything that surprises you or contradicts an assumption you had

Tip: You do not need to memorize anything. You are building intuition so the next two stages feel less abstract.

4. Now run /eval-suite-planner to create your eval plan. Describe your agent in 2-3 sentences:

/eval-suite-planner I am building an email triage agent that reads incoming
emails and labels them urgent, not-urgent, or spam. It should be polite and
never label a real customer email as spam.

You'll get a scenario plan table with named scenarios, evaluation methods, and thresholds. Do not close this session — the generator in Stage B will read this plan.

Checkpoint

Before moving to Stage B, you should have: (1) a scenario plan table in your conversation, and (2) be able to answer in one sentence: "For my chosen agent, the most important thing to measure is ___."

Stage B — Generate Scenarios and Write Your Eval File

Time box: 15 minutes (0:30–0:45)

This is the main building block. You will pick an agent type, generate eval scenarios with AI assistance, save a dataset, and wire up an eval file.

Step 1 — Choose your agent task

Pick one of the four options below. If you have a real agent you work on, use that instead and adapt the instructions accordingly.

Option Agent description
Email triage Reads incoming emails and labels them as urgent, not urgent, or spam
Code review Reads a code diff and identifies potential bugs or style issues
Meeting notes Takes a raw meeting transcript and produces a structured summary with action items
Product description Takes a product name and specs and writes a short marketing blurb

Write down your choice. You will reference it in every prompt you send.

Step 2 — Generate scenarios with /eval-generator

2a. In the same Claude Code session from Stage A (where you ran the planner), run /eval-generator. It will automatically detect the planner's output and generate one test case per scenario:

/eval-generator

If you're in a new session (no planner output available), provide your agent description directly:

/eval-generator I am building a meeting notes agent.
It takes a raw meeting transcript and produces a structured summary with action items.

2c. Wait for the output. A good response includes: - At least 5–6 distinct scenario descriptions - A mix of case types: one or two happy-path, one or two edge cases (empty input, very long input, ambiguous content), one adversarial case (input designed to trip the agent up) - For each scenario: a description, a sample input, and a suggested expected outcome or criterion

Tip: If the output feels too generic, run it again with a more specific prompt. Add a sentence like: "Focus on cases where the input is ambiguous or the output format matters."

Step 3 — Review and edit the generated scenarios

3a. Read through each generated scenario. Ask yourself: - Is this realistic? Would this actually happen in production? - Is the expected outcome specific and testable, or vague? - Is there an important scenario that is missing?

3b. Delete any scenario that is too similar to another one. You want variety, not duplicates.

3c. Write one scenario yourself — one that the AI did not generate. This should be a case you personally think is important. Add it to your list.

Common mistake: Expected outcomes like "the summary should be good" are not testable. Rewrite them as specific constraints: a maximum length, a required topic, a required format, or a prohibited phrase.

Step 4 — Save your dataset file

4a. Create the dataset file. The harness expects files in my-evals/datasets/:

mkdir -p my-evals/datasets
cp examples/datasets/summarizer-cases.json my-evals/datasets/my-eval-cases.json

4b. Open my-evals/datasets/my-eval-cases.json and replace all of its contents with your own cases (copy the JSON from the generator output). Use this format for each case:

[
  {
    "id": "case-001",
    "input": {
      "text": "Paste your sample input here."
    },
    "expected": {
      "max_length": 300,
      "must_include_topic": "action items"
    },
    "tags": ["happy-path"]
  },
  {
    "id": "case-002",
    "input": {
      "text": "Paste an edge-case input here."
    },
    "expected": {
      "max_length": 300,
      "must_include_topic": "action items"
    },
    "tags": ["edge-case"]
  }
]

Aim for at least three cases total: one happy-path, one edge case, one adversarial or tricky case.

Common mistake: JSON does not allow trailing commas. A comma after the last item in an array or object will cause a parse error. Double-check before saving.

Step 5 — Copy and edit the eval template

5a. Copy the reference eval file into my-evals/.

cp examples/summarizer-eval.ts my-evals/my-agent-eval.ts

5b. Open my-eval.ts. Make the following five edits:

Edit 1 — Dataset path. Find the line that loads the dataset and change the path to your file.

// Change this:
const dataset = loadDataset("examples/datasets/summarizer-cases.json");

// To this:
const dataset = loadDataset("my-evals/datasets/my-eval-cases.json");

Edit 2 — Eval name. Update the eval name so results are easy to identify.

// Change this:
name: "summarizer-eval",

// To something like:
name: "meeting-notes-eval",

Edit 3 — System prompt. Replace the summarizer system prompt with a prompt for your agent. Write it in plain English: what is the agent's job, what format should it produce, what should it never do?

Edit 4 — Input template. Update the template variable that maps input.text into the prompt. If your dataset uses a different input field name, match it here.

Edit 5 — Grader criteria. Update the grader to reflect your expected outcomes. If you are using a deterministic grader, check that the field names (like max_length or must_include_topic) match what you put in your dataset's expected block. If you are using a model-graded grader, rewrite the criteria string to describe what "good" looks like for your agent.

Tip: You do not need to write a perfect grader. Get it working first, then refine it. A grader that is slightly wrong is more useful than no eval at all.

Checkpoint

Before moving to Stage C, verify: - my-evals/datasets/my-eval-cases.json exists and is valid JSON (no trailing commas) - my-evals/my-agent-eval.ts compiles without errors: npx tsc --noEmit my-evals/my-agent-eval.ts - You can describe in one sentence what your grader is checking

Stage C — Run the Eval and Interpret Results

Time box: 10 minutes (0:40–0:50)

Step 1 — Run your eval

npm run eval -- --file my-evals/my-agent-eval.ts --verbose

Watch the output as it runs. You will see each case execute and a pass/fail result per case. The --verbose flag shows full model output so you can read what the agent actually said.

Step 2 — Interpret with /eval-result-interpreter

Copy the terminal summary (pass/fail counts and any failure details) and paste it into the interpreter:

/eval-result-interpreter 5/8 passed. Failed: case-003 (topic "refund"
missing), case-006 (length 520/400), case-008 (adversarial safety failed)

The interpreter will give you a SHIP / ITERATE / BLOCK verdict, classify each failure by root cause (eval setup issue, agent config issue, or platform limitation), and give you the top 3 actions to take.

Alternative — Copilot Studio path: If you ran your eval in Copilot Studio and exported a CSV, provide the file path instead:

/eval-result-interpreter C:\Users\you\Downloads\Evaluate Agent.csv

Step 3 — Answer the three debrief questions

Write down your answers before the group debrief at 0:50. You will be asked to share.

  1. Which cases passed? Which failed?
  2. For each failure: is this an eval setup issue (the test is wrong) or an agent issue (the agent needs fixing)? The interpreter classifies this for you, but form your own opinion too.
  3. What is the #1 thing you would change before re-running?

Tip: A failing case is not a bad outcome. It means you found signal. Per the Triage Playbook, at least 20% of failures in a new eval are eval setup issues, not agent issues — always check the eval first.

Troubleshooting

Invalid JSON error when running the eval Almost always a trailing comma. Open datasets/my-eval-dataset.json, look for a comma on the last line before a ] or }, and remove it. Use a JSON validator if needed: paste your content into jsonlint.com.

File not found / path mismatch Check that the path in my-eval.ts exactly matches where you saved the file. Paths are relative to the harness root directory. Run ls datasets/ to confirm the file exists.

TypeScript compilation error Run npx tsc --noEmit my-eval.ts to see the exact error. Most common cause: a field name in the grader does not match a field name in the expected block of the dataset. Check spelling and casing.

Auth error when running /eval-faq or /eval-generator Your Claude Code session may have timed out. Run claude auth status and re-authenticate if needed, then re-run the slash command.

npm run eval — cannot find module You are likely running the command from the wrong directory. Navigate to the harness root (the directory containing package.json) and try again.

cd /path/to/eval-harness
npm run eval -- --file my-eval.ts

Stretch Goals

Finished early? Try one or more of these extensions.

Add a second grader. Your eval file likely has one grader. Add a second one that checks something different — for example, if your first grader checks length, add a model-graded grader that checks tone or completeness. Run the eval again and see if the results change.

Add harder cases. Go back to datasets/my-eval-dataset.json and add two adversarial cases: one where the input is malformed or empty, and one where the input is designed to produce an output that looks correct but is actually wrong. Re-run and see how your agent handles them.

Filter by tag. Add a "tags": ["adversarial"] field to your hardest cases, then run:

npm run eval -- --file my-eval.ts --tags adversarial

Compare the pass rate on adversarial cases versus happy-path cases.

Use verbose mode. Run with --verbose to see the full model output for each case:

npm run eval -- --file my-eval.ts --verbose

Read the actual outputs. Do they look right to a human, even when the grader marks them as failing? If yes, your grader criteria may be too strict.

Go back to /eval-faq with a specific result. Take a failing case and ask:

/eval-faq My knowledge grounding grader is failing on cases where the
input contains multiple topics. What might be causing this?

Use the answer to refine either your grader or your dataset.
