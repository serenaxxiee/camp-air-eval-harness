Camp AIR: AI Agent Evaluation Best Practices

1-Hour Workshop — Facilitator Session Outline

Session at a Glance

Segment Start End Duration Format
Opening / Framing 0:00 0:05 5 min Presentation
Core Concepts 0:05 0:15 10 min Presentation + Q&A
Live Demo 0:15 0:25 10 min Facilitator demo
Hands-On Exercise 0:25 0:50 25 min Participant work
Debrief & Wrap-Up 0:50 1:00 10 min Group discussion

Total: 60 minutes

Participant-to-facilitator ratio: up to 30:1 recommended. One roving assistant is ideal but not required.

Prerequisites

Participants must complete these steps before arriving. Send this list in the pre-session email at least 24 hours in advance.

Required: - Node.js 18+ installed (node --version to confirm) - Claude Code CLI installed and authenticated (claude --version to confirm) - The eval harness repository cloned locally: git clone https://github.com/camp-air/eval-harness cd eval-harness npm install - Two Claude Code slash command skills installed: - /eval-faq — answers eval methodology questions - /eval-generator — generates eval scenarios from a task description

Recommended: - A code editor open and ready (VS Code preferred) - Basic familiarity with TypeScript (participants don't need to write it from scratch, but should be able to read it) - Have run at least one AI agent before — participants should know what a prompt is and roughly what an LLM call looks like

Not required: - Prior experience writing evals - Deep TypeScript knowledge - Any specific domain knowledge

Room / Environment Setup (Before Participants Arrive)

  • Project the eval harness repo on screen with a terminal split alongside
  • Have the /eval-faq and /eval-generator slash commands pre-installed in your demo environment
  • Run npm test once ahead of time to confirm the harness works end-to-end
  • Prepare a working example eval (the examples/summarizer-eval.ts file in the repo works well)
  • Have a shared doc or whiteboard ready for the debrief — participants will paste results into it
  • If using a shared WiFi network, confirm npm install works or have node_modules cached

Segment 1: Opening / Framing

Time: 0:00 – 0:05 (5 minutes)

Learning objective: Participants understand why evals are not optional and feel the urgency of building them early, not late.

Facilitator Talking Points

Open with this question — say it out loud and pause for 3–4 seconds before continuing:

"How many of you have shipped an AI feature and later found out it was silently broken for a week before anyone noticed?"

Wait for hands or nods. Most audiences will respond.

Continue:

"That's the eval problem. When you don't have automated evals, you're flying blind. You're relying on someone noticing that the AI is doing something wrong. And in production, that often means a user noticed first."

Then frame what evals actually are — not a QA afterthought, but an engineering discipline:

"Evals are how you measure whether your AI is doing what you intend. They're automated tests, but the assertion logic is different — instead of checking if a value equals another value, you're checking if a response is good, grounded, safe, helpful, or whatever your success criteria are."

Key framing to land before moving on:

  • Evals are to AI agents what unit tests are to software — except harder, because the output space is open-ended
  • Without evals, you cannot iterate confidently — every change to a prompt is a gamble
  • With evals, you can move fast without breaking things

Set the agenda for the session. Walk through the session-at-a-glance table briefly — 30 seconds max. Participants should know they will write and run a real eval before they leave.

Timing Notes

  • Do not let the opening run over 5 minutes. The hook lands fast or not at all.
  • If the audience is quiet after the opening question, move on immediately — do not wait for verbal responses.
  • End this segment by saying: "Let's get grounded in the vocabulary first, then I'll show you a working example."

Segment 2: Core Concepts

Time: 0:05 – 0:15 (10 minutes)

Learning objective: Participants can define and distinguish: prompt, criteria, grader, dataset, and harness — and explain how they fit together in a single eval run.

Facilitator Talking Points

Walk through each term. Write them on the whiteboard or show a slide as you go. The goal is shared vocabulary, not deep theory.

Prompt

"The input to your AI system. In evals, you're not testing one prompt — you're testing a prompt template against many inputs. The template stays fixed; only the data changes."

Example to say aloud:

"If you're building a customer support agent, your prompt template might be: 'You are a helpful support agent. Answer this customer question: {{question}}'. The eval swaps in 20 different questions and checks the outputs."

Criteria

"What does 'good' look like for this task? Criteria make your success definition explicit and testable. They should be specific enough that two people would agree on a pass/fail judgment."

Good criteria examples: - "The response does not mention competitor products by name" - "The summary is 3 sentences or fewer" - "The response answers the user's question without asking for clarification"

Bad criteria examples (too vague): - "The response is good" - "The answer is helpful"

Grader

"The thing that applies your criteria to a response. Graders come in three flavors."

List the three types:

  1. Deterministic / rule-based — regex, string matching, JSON schema validation. Fast, cheap, reliable. Use these first.
  2. Model-graded — you send the response to an LLM with your criteria and ask it to judge. Flexible, handles nuance, but adds cost and latency.
  3. Human — a person reviews the output. Slowest, most expensive, highest quality. Use for calibration and edge-case triage.

"In practice, you layer these. Fast rule-based graders run first and catch obvious failures. Model-graded graders handle the judgment calls."

Dataset

"The collection of inputs your eval runs against. A dataset is not a single test case — it's a representative sample of what your system will face in production."

Key points: - Small is okay to start. 10–20 well-chosen cases beat 200 random ones. - Datasets should include happy-path cases, edge cases, and adversarial cases. - Your dataset grows over time — every time a user hits an unexpected behavior, that's a new dataset entry.

Harness

"The scaffolding that orchestrates the whole thing. A harness loads your dataset, runs each input through your system, applies your graders, and produces a report. It's the thing you run in CI."

Draw a simple flow on the board:

Dataset -> [for each input] -> System (prompt + model) -> Grader -> Results

"Everything we're doing today lives inside this loop."

Q&A Pause

After covering all five terms, pause and ask:

"Any of these feel unclear before we see them in action?"

Take 1–2 questions maximum. If a question goes deep, say: "Great question — let's see if it comes up in the demo and we can address it then." Keep the clock moving.

Timing Notes

  • Budget roughly 2 minutes per concept. Do not go deeper on any single term — the demo will make them concrete.
  • If you're running long, cut the Q&A pause. The demo is more valuable than extra explanation time.
  • Transition line: "Alright — let's stop talking about evals and look at one actually running."

Segment 3: Live Demo

Time: 0:15 – 0:25 (10 minutes)

Learning objective: Participants see a complete eval cycle — from dataset to scored results — and can map what they see back to the five concepts from Segment 2.

Setup

Before starting the demo, have these ready on screen: - Terminal at the root of the eval harness repo - examples/summarizer-eval.ts open in your editor - The /eval-generator slash command available in Claude Code

Demo Script

Step 1 — Show the dataset (1 min)

Open examples/datasets/summarizer-cases.json. Scroll through it briefly.

Say:

"Here's our dataset. Five inputs — each one is an article snippet the agent will summarize. Notice we've also got an expected length and a topic label per case. Those will feed into our graders."

Step 2 — Show the eval file (2 min)

Open examples/summarizer-eval.ts. Walk through it top to bottom, pointing out:

  • Where the dataset is loaded
  • The prompt template
  • The two graders: a character-count check (deterministic) and a model-graded relevance check
  • Where results are written out

Say:

"This is the whole thing. It's about 60 lines. The harness handles the loop — you just define the dataset, the prompt, and the graders."

Step 3 — Run it (2 min)

In the terminal, run:

npm run eval -- --file examples/summarizer-eval.ts

Let it run. Narrate what's happening:

"You'll see it process each case in sequence. The deterministic grader runs instantly. The model-graded grader makes an LLM call — that's where the latency is."

When it finishes, show the output table in the terminal.

Step 4 — Interpret the results (2 min)

Point at the results. Pick one that passed and one that failed.

Say:

"Case 3 failed the length check — the summary was 412 characters, and our threshold was 400. That's a hard failure. Case 4 passed both graders. Case 2 passed the length check but got a 0.6 on relevance — borderline. We'd want to look at that one manually."

This is a key teachable moment:

"Notice that failing an eval doesn't mean your AI is broken. It means you have information. Maybe the length threshold is wrong. Maybe the model is genuinely off. The eval gives you something to act on."

Step 5 — Show eval-generator (2 min)

Open Claude Code and run:

/eval-generator

Prompt it with:

"Generate eval scenarios for a customer support agent that handles refund requests. The agent should be polite, follow the refund policy, and not make promises the policy doesn't allow."

Show 3–4 of the generated scenarios. Point out that each one has an input, an expected behavior, and a suggested scoring criterion.

Say:

"This is how AI accelerates the eval-writing process. You describe the task and the success criteria in plain language, and the tool scaffolds your test cases. You review, edit, and add them to your dataset. You're not starting from a blank page."

Timing Notes

  • This demo must stay at 10 minutes. Practice it ahead of time with a timer.
  • If the model-graded grader is slow (>90 seconds), skip the live run and show a pre-recorded output — have a results-snapshot.txt file ready as backup.
  • Do not get pulled into the TypeScript details. If someone asks about a specific line of code, say: "Good catch — you can dig into that during the hands-on section."
  • Transition line: "Now it's your turn. You're going to write and run your own eval in the next 25 minutes."

Segment 4: Hands-On Exercise

Time: 0:25 – 0:50 (25 minutes)

Learning objective: Participants write a working eval — dataset, prompt, at least one grader — and run it against the harness. They observe and interpret their own results.

Overview

This is the core of the session. Participants work individually or in pairs. The exercise has three stages that build on each other.

Stage A — Use /eval-faq to get oriented (5 min) Stage B — Use /eval-generator to create scenarios, then write an eval (15 min) Stage C — Run the eval and interpret results (5 min)

Stage A: Use eval-faq (0:25 – 0:30, 5 min)

Facilitator action: Give these instructions verbally and display them on screen.

Say:

"First, open Claude Code and run the /eval-faq command. Ask it at least two questions about eval methodology. This is your onramp — use it to get clarity on anything from the concepts segment that wasn't fully clear."

Suggested questions participants can ask (display these on a slide): - "What makes a good eval dataset?" - "When should I use a model-graded grader vs. a rule-based one?" - "How do I write criteria that are specific enough to be testable?" - "What's the minimum viable eval I can write in 15 minutes?" - "How do I know if my eval is actually measuring what I think it is?"

Facilitator action during Stage A: Walk the room. Glance at what people are asking. If someone has a great question/answer exchange, make a mental note — you may call on them in the debrief.

Timing note: This stage is timeboxed at 5 minutes. At the 5-minute mark, give a verbal prompt: "Okay, wrap up your eval-faq questions — we're moving to scenario generation."

Stage B: Generate Scenarios and Write an Eval (0:30 – 0:45, 15 min)

Facilitator action: Give participants their task. Display the task prompt on screen.

The Task (display verbatim):

Choose one of the following agent tasks — or propose your own if you have something in mind:

  1. Email triage agent — reads incoming emails and labels them as urgent / not urgent / spam
  2. Code review agent — reads a code diff and identifies potential bugs or style issues
  3. Meeting notes agent — takes raw meeting transcript and produces a structured summary with action items
  4. Product description agent — takes a product name and specs and writes a short marketing blurb

Then: 1. Run /eval-generator in Claude Code and describe your chosen agent task. Ask it to generate 5–8 eval scenarios. 2. Review the generated scenarios. Edit or discard any that don't make sense. Add one scenario of your own. 3. Save your scenarios to a file called my-eval-dataset.json in the harness datasets/ directory. 4. Open examples/summarizer-eval.ts as a starting template. Copy it to my-eval.ts. 5. Edit my-eval.ts to: - Point to your dataset - Update the prompt template to match your agent task - Add or modify at least one grader that reflects your criteria

What facilitators should watch for during Stage B:

  • Participants who are stuck on the TypeScript — point them to examples/summarizer-eval.ts and tell them to change as little as possible. The goal is one working eval, not a perfect one.
  • Participants who finish early — prompt them to add a second grader (if they only have one) or add 3 more dataset cases.
  • Participants who are generating scenarios but not editing them — encourage them to delete at least one generated scenario and write a replacement. This builds the habit of reviewing AI-generated test cases rather than accepting them wholesale.
  • Common confusion: participants sometimes conflate the grader and the criteria. Remind them: the criteria is the rule in plain English; the grader is the code or model call that applies it.

Facilitator talking point to drop mid-stage (around the 10-minute mark):

Walk to the center of the room and say:

"Quick check-in — raise your hand if you've got your dataset file saved and you're working on the eval file now."

Count hands. If fewer than half have raised their hands, extend Stage B by 3 minutes and compress Stage C. If most are there, keep the pace.

Stage C: Run the Eval and Interpret Results (0:45 – 0:50, 5 min)

Facilitator action: Give the run instruction verbally and display on screen.

Say:

"Time to run it. In your terminal, from the harness root, run:" npm run eval -- --file my-eval.ts "When it finishes, look at your results and answer these three questions — you'll share them in the debrief:" 1. Which cases passed? Which failed? 2. Does the failure make sense, or does it look like your grader is wrong? 3. What's one thing you'd change — in the prompt, the dataset, or the grader — based on what you saw?

Facilitator action: Walk the room and look at results. Take note of 2–3 interesting or contrasting outcomes to highlight in the debrief.

If a participant's eval throws an error: - Check that the dataset JSON is valid (common issue: trailing commas) - Check that the file path in the eval matches the dataset filename exactly - If it's a TypeScript compilation error, have them compare their file against examples/summarizer-eval.ts line by line

Timing note: At 0:49, give a verbal 1-minute warning: "One more minute — wrap up what you're looking at, we're going to share results."

Timing Contingency

If Stage B runs long and participants don't finish their eval by 0:45, shift Stage C to: "Run the eval, even if it's incomplete. A partial run is fine — look at whatever output you get and bring that to the debrief." Do not skip Stage C entirely — running the harness at least once is a core outcome of the session.

Segment 5: Debrief & Wrap-Up

Time: 0:50 – 1:00 (10 minutes)

Learning objective: Participants can articulate one concrete change they'd make to their eval and leave with a clear next step for applying evals to their own projects.

Debrief (0:50 – 0:57, 7 min)

Facilitator action: Bring the group back together. If there's a shared doc or whiteboard, ask participants to paste or write their three answers from Stage C.

Work through the room by calling on 3–4 people (or pairs). For each one, ask:

  1. "What task did you pick?"
  2. "What did your results look like — mostly passing, mostly failing, mixed?"
  3. "What would you change based on what you saw?"

Talking points to weave into the debrief discussion:

On unexpected failures:

"If your grader is flagging things that look correct to you, that's actually the most valuable outcome of this exercise. It means your criteria aren't precise enough yet — and the eval is telling you that before production does."

On unexpected passes:

"If everything passed on the first run, add a harder case. Evals that never fail aren't protecting you — they're just not covering the hard cases yet."

On the eval-generator output:

"How many of you edited or deleted at least one scenario the generator gave you?"

Follow up with whoever raises a hand:

"What made you change it?" — This surfaces the key lesson: AI-generated scenarios are a starting point, not a finished artifact. Reviewing them is part of the process.

On the AI-first workflow:

"Think about what just happened. You described a task in plain English, a tool gave you test scenarios, you ran them through a harness, and you have structured results to act on. That whole loop took 20 minutes. Without the AI tools, writing that dataset from scratch would have taken an afternoon. The velocity gain is real — but only if you stay in the loop as the reviewer."

Key Takeaways (0:57 – 1:00, 3 min)

Read these aloud and display them on screen. Participants should leave with these five things in their heads.

1. Start with criteria, not code. Before writing a single line of eval code, write down what "good" means in plain English. Specific, testable criteria are the foundation everything else rests on.

2. Small datasets are fine. Representative datasets are essential. 10 well-chosen cases that cover your real input distribution are worth more than 100 randomly generated ones. Include happy paths, edge cases, and at least one adversarial case.

3. Layer your graders. Start with fast, deterministic checks. Add model-graded scoring for the judgment calls. Save human review for calibration and triage — not for every run.

4. AI accelerates the scaffolding, not the thinking. Use /eval-generator to get scenarios quickly. Use /eval-faq to get unstuck. But you still have to define the criteria, review the cases, and interpret the results. The thinking stays with you.

5. Evals are living artifacts. A dataset you write today and never touch again will rot. Every production incident, every surprising model output, every edge case a user hits — these are all new dataset entries. Build the habit of growing your eval suite alongside your product.

Closing

End with this line:

"The best time to write an eval is before you write the prompt. The second best time is right now. You've all written one today — keep that file. That's your foundation."

Thank participants. Point them to: - The eval harness repo (for reference and the example files) - The /eval-faq slash command (available any time in Claude Code) - The /eval-generator slash command (for their next project)

Facilitator Reference: Common Questions and How to Handle Them

These are real questions that come up in this session. Have answers ready.

"How do I know how many eval cases I need?" Start with enough to cover your input distribution — typically 15–30 for a focused task. If you're building a classifier, you need representation of each class. If you're building a summarizer, you need short inputs, long inputs, ambiguous inputs. The number matters less than the coverage.

"Can I use this for multi-turn agents, not just single-turn?" Yes, but it's harder. The harness supports multi-turn via conversation fixtures — see examples/multi-turn-eval.ts. The scoring logic stays the same; you're just scoring the final turn or a specific intermediate turn.

"Isn't model-graded scoring circular? The AI grading itself?" It's a fair concern. The grader model and the system-under-test don't have to be the same model — and usually shouldn't be. Also, the grader has a much simpler job: it's applying specific criteria to a specific output, which is far easier than generating the output itself. Treat grader scores as noisy labels, not ground truth. Calibrate them against human judgments on a sample.

"What do I do when my grader disagrees with my gut?" Trust the disagreement. Either your grader is wrong (the criteria are imprecise) or your gut is wrong (you have a blind spot). In both cases, the disagreement is signal. Investigate it, don't paper over it.

"How do I run evals in CI?" The harness supports a --ci flag that exits with a non-zero code if any grader falls below a threshold. Plug that into your GitHub Actions or equivalent. See docs/ci-setup.md in the repo for a step-by-step.

"How much does model-graded scoring cost?" For a 20-case eval with one model-graded grader, you're looking at roughly 20 extra LLM calls. At typical API pricing, that's cents per run. The bigger concern is latency — each call adds 1–3 seconds. If you're running evals frequently, consider batching grader calls or caching results for identical inputs.

Appendix: Sample Dataset Format

Participants can reference this if they're unsure how to format my-eval-dataset.json.

[
  {
    "id": "case-001",
    "input": {
      "text": "The quarterly revenue was $4.2M, up 18% from last quarter. Key drivers were enterprise deals and reduced churn."
    },
    "expected": {
      "max_length": 400,
      "must_include_topic": "revenue"
    },
    "tags": ["happy-path", "financial"]
  },
  {
    "id": "case-002",
    "input": {
      "text": "Meeting adjourned."
    },
    "expected": {
      "max_length": 100,
      "must_include_topic": null
    },
    "tags": ["edge-case", "minimal-input"]
  }
]

Fields: - id — unique identifier for the case. Used in result output. - input — the data passed to your prompt template. Shape depends on your template. - expected — the values your graders will check against. Can hold any fields your graders reference. - tags — optional. Useful for filtering results by category.

Appendix: Eval Harness Quick Reference

npm run eval -- --file <path-to-eval-file>     # Run a specific eval
npm run eval -- --file <path> --ci             # CI mode: exits non-zero on failure
npm run eval -- --file <path> --tags edge-case # Run only cases with a specific tag
npm run eval -- --file <path> --verbose        # Show full model outputs in terminal

Results are written to results/<eval-filename>-<timestamp>.json after each run.
